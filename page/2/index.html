符号解析

符号解析就是链接器将“引用的符号”与“定义的符号”进行匹配的过程。

链接器如何解析多重定义的全局符号

链接器处理多个 可重定位目标文件（.o 文件）。

每个模块中可能会定义同名的全局符号（如变量、函数）。

符号分为两类：

  • 强符号（strong）：函数、已初始化的全局变量（如 int x = 1;）
  • 弱符号（weak）：未初始化的全局变量（如 int x;）
规则编号 描述
规则 1 多个强符号同名 → 报错，链接失败（冲突无法解决）。
规则 2 强符号 vs 弱符号同名 → 选择强符号，忽略弱的。
规则 3 多个弱符号同名 → 任意选择一个（通常选择第一个出现的）。

静态库

静态库（static library）：是由多个可重定位目标文件（.o）打包成的一个 .a 文件，供链接器使用。

链接

链接（Linking）是一个将多个代码和数据片段（来自编译器生成的目标文件）收集并组合为一个单一文件的过程，这个文件可以被加载到内存中执行。

链接的目标文件可以是：

  • 可执行文件
  • 共享库（动态库）
  • 或中间的可重定位文件

链接可以发生在不同的阶段：

链接阶段 说明
链接时（Link-time） 最常见的，指用 ld 或 gcc 把多个 .o、.a、.so 链接为一个可执行文件或库。
加载时（Load-time） 程序被操作系统加载到内存时，动态链接器（如 /lib64/ld-linux.so.2）把 .so 文件装载并绑定符号。
运行时（Run-time） 程序运行过程中自行调用 dlopen() / dlsym() 等函数动态加载库并解析符号。

静态链接

静态链接是指链接器将所有需要的代码、数据和库函数一次性合并到一个最终的可执行文件中，使这个文件在运行时不再依赖外部库

struct sock

发表于 2025-06-22 分类于 网络协议栈源码学习 Valine：

sock 结构体在 Linux 网络协议栈中的作用非常核心，它本质上是用于管理传输层连接状态和控制信息的内核数据结构

sock 结构体的主要功能：

1. 管理传输层连接状态

  • 记录 socket 状态，如 TCP 的 ESTABLISHED、LISTEN、SYN_SENT 等；
  • 维护连接生命周期。

2. 数据收发管理

  • 接收队列：sk_receive_queue
  • 发送队列：sk_write_queue
  • Backlog 队列：sk_backlog

3. 缓冲区和流控

  • 接收缓冲区大小（sk_rcvbuf）
  • 发送缓冲区大小（sk_sndbuf）
  • 控制数据收发速率，避免内存耗尽。

4. 定时器与超时管理

  • sk_timer：重传、超时的定时器；
  • 如 TCP 的重传定时器（RTO）、保活定时器（keepalive）。

处理器体系结构

一个处理器支持的指令和指令的字节级编码称为它的指令集体系结构(Instruction-Set Architecture, ISA)

本章将简要介绍处理器硬件的设计，学习一个硬件系统执行某种 ISA指令的方式，以更好地理解计算机是如何工作的。

本章首先定义一个简单的指令集，作为处理器实现的运行示例。因为受 x86-64 指令集的启发，它被俗称为 “x86”, 所以我们称我们的指令集为 “Y86-64” 指令集。 与 x86-64 相比， Y86-64 指令集的数据类型、指令和寻址方式都要少一些。它的字节级编码也比较简单，虽然 Y86-64 指令集很简单，它仍然足够完整，能让我们写一些处理整数的程序。

Y86-64 的寄存器

  • 寄存器：Y86-64 有 15 个通用寄存器（与 x86-64 类似），如 %rax、%rsp、%rbp 等，每个存储 64 位数据。%rsp 主要用作栈指针。

  • 条件码：ZF、SF、OF 用于保存最近计算的结果信息。

  • 程序计数器（PC）：存储当前将要执行的指令地址。

Y86-64 指令

  • y86-64 的 movq 指令分成了 4 个不同的指令：irmovq、rrmovq、mrmovq 和 rmmovq，分别显示地指明源和目的的格式。源可以是立即数（i）、寄存器（r）或内存（m）。指令名字的第一个字母就表明了源的类型，目前可以是寄存器（r）或内存（m）。指令名字的第二个字母指明了目的的类型。它决定如何实现数据传送时，显式地指明数据传送的路径。
  • 有 4 个整数操作指令，如图它们是 addq、subq、andq 和 xorq。它们只对寄存器数据进行操作，而 Y86-64 还允许对内存数据进行这些操作。这些指令会设置 3 个条件码 ZF、SF 和 OF（零、符号和溢出）。
  • 7 个跳转指令是 jmp、jle、jl、je、jne、jge 和 jg。根据分支指令的类型和条件代码的设置来选择分支。分支条件和 x86-64 的一样）。
  • 有 6 个条件传送指令：cmovle、cmovl、cmove、cmovne、cmovge 和 cmovg。这些指令的格式与寄存器-寄存器传送指令 rrmovq 一样，但是只有当条件码满足所需的要求时，才会更新目的寄存器的值。
  • call 指令将返回地址入栈，然后跳到目的地址。ret 指令从栈中调用出返回。
  • pushq 和 popq 指令实现了入栈和出栈，和 x86-64 中一样。
  • Y86-64 只有一条与主机操作相关的指令 halt。x86-64 的应用程序不允许使用这条指令，因为它会导致整个系统停止运行。对于 Y86-64 来说，执行 halt 指令会导致处理器终止，并将状态设置为 HLT。

数据对齐

数据按对齐要求存放，可以让处理器和内存系统更高效地访问数据

对齐的要求

比如 64 位系统（x86-64），一个 double 或 long 类型的变量通常要求8字节对齐，也就是说，变量的地址必须是8的倍数。

下表格列出了常见类型的对齐要求（K）：

K 类型

2 short
4 int, float
8 long, double, char*

函数调用

函数调用是一种代码封装机制，他通过参数和返回值实现特定的功能，隐藏实现的细节，提供清晰的接口

实现函数调用需要提供以下机制：

  • 控制转移：跳转到被调用过程并在结束后返回原处；
  • 参数传递：将调用方的数据传给被调用方；
  • 返回值传递：将结果从被调用方传回；
  • 局部变量管理：分配并释放被调用过程所需的内存

x86-64 架构的过程调用依赖一套特定的寄存器和内存使用规范，以减少开销和提高效率。

栈帧

栈帧就是每次函数调用时，在栈上分配的一块内存区域,里面通常会包含：

内容 作用
参数区 用来保存传入函数的参数（如果没用寄存器传）
返回地址 函数执行完后，跳回调用者的地方
保存的寄存器 保存调用者的寄存器内容，防止被破坏
局部变量 函数内部使用的变量（如 int x = 5;）
临时空间 编译器优化、对齐等需求
