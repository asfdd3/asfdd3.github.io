路由相关数据结构

引言

在 Linux 6.6 内核中，路由子系统不再依赖早期的缓存机制，而是以 fib_trie 前缀树为核心，实现高效的路由查找。整个子系统围绕“从策略到前缀匹配，再到下一跳决策”这一主线展开：入口由策略路由规则（fib_rule）决定查询路径，路由表（fib_table）承载具体数据，内部通过压缩前缀树（fib_trie / key_vector）完成最长前缀匹配；同一前缀下的多条路由以 fib_alias 链表组织，并通过 fib_info 进行去重和共享，最终由 fib_nh 描述下一跳信息，生成 rtable 作为查找结果供协议栈使用。相关的关键数据结构如下：

IP分片

发表于 2026-03-12 更新于 2026-03-15 分类于 网络协议栈源码学习 Valine：

在网络中，一个报文从源主机发送到目的主机的过程中，可能会经过多条不同类型的网络链路，而这些链路所能承载的最大传输单元MTU并不一定相同。当一个 IP报文的大小超过某条链路允许的 MTU 时，如果直接发送将无法通过该链路。为了解决这一问题，IPv4 协议设计了 IP 分片机制：在必要时将一个较大的 IP 数据报拆分成多个较小的分片分别传输，每个分片都携带自己的 IP 头部信息，并在最终目的主机处重新组装成原始数据报。在这一过程中，IP 头中的 Identification、Fragment Offset 以及 MF（More Fragments）标志位 用于标识分片之间的关系和顺序，从而保证数据能够正确地重组。

内核源码中对应处理IP分片的入口是ip_fragment当，数据包的长度大于MTU可以承载的大小时，会调用该函数完成分片具体代码如下所示：

快速路径

快速路径 的核心思想是如果原始 skb 已经以 frag_list 的形式组织了多个子 skb（通常来自上层构造的分段结构），并且这些子 skb 的大小、8 字节对齐、头部空间以及引用状态都满足分片要求，那么内核就不再重新复制数据进行分片，而是直接复用这些已有的子 skb 作为 IP 分片。具体流程是：首先检查第一个 skb 和所有 frag_list 中的 skb 是否满足 长度 ≤ mtu、非克隆、8 字节对齐、头部空间足够 等条件，如果全部合法，就通过 ip_fraglist_init() 初始化遍历状态，然后在循环中依次为每个子 skb 调整 IP 分片字段（frag_off、MF、offset 等），随后直接调用 output() 将这些 skb 逐个发送。整个过程中不需要重新分配 skb 或复制 payload，只是修改头部并发送，从而避免数据拷贝，提高分片性能。

IP选项处理（一）

发表于 2026-03-05 更新于 2026-03-04 分类于 网络协议栈源码学习 Valine：

引言

在 IPv4 协议设计之初，协议头部被设计为“固定部分 + 可选扩展”的结构。固定首部长度为 20 字节，而当 IHL（Internet Header Length）字段大于 5 时，表示在固定首部之后还存在可选字段（IP Options）。这些选项最多可以扩展 40 字节，使 IPv4 首部最大达到 60 字节。

IP Options 的设计初衷是增强网络层的可观测性与可控性，例如记录路径、指定路由、测量时延等。但随着互联网规模扩大以及安全问题凸显，这些机制逐渐被认为会带来性能下降与安全风险。现代网络中，绝大多数 IPv4 报文都不携带任何选项，而带有 Options 的报文通常会触发路由器或操作系统的慢路径处理。

选项结束（End of Option List）

核心作用： 标记 IP 选项列表结束。
使用现状： 仍可能出现（仅用于填充）。
说明： 占 1 字节，用于提前终止选项解析或做对齐填充，不承载实际网络语义。

空操作（No Operation）

核心作用： 用于选项字段对齐。
使用现状： 仍可能出现（用于对齐）。
说明： 占 1 字节，通常用于将后续选项对齐到 4 字节边界，本身不具备功能性含义。
