igb 网卡驱动发送路径

发表于 2026-06-30 更新于 2026-06-29 分类于 IGB网卡驱动 Valine：

前面已经分析了 igb 网卡驱动的初始化流程，包括设备探测、资源申请、队列初始化以及 open 阶段对硬件寄存器的配置。网卡完成初始化之后，真正的数据发送入口就是 ndo_start_xmit 回调。在 igb 驱动中，这个回调对应的是 igb_xmit_frame() 函数。协议栈在需要发送数据包时，会把构造好的 skb 交给该函数，由驱动完成 TX 队列选择、描述符填充、DMA 映射以及通知硬件发送等工作。本文从 igb_xmit_frame 这一入口开始，梳理 igb 网卡驱动的发送路径。

igb_xmit_frame

这段代码就是 igb 驱动的发包总入口：先从 netdev 找到网卡私有结构 adapter，然后确保待发送的 skb 至少有 17 字节，满足硬件最小发送长度要求，最后根据 skb 选择对应的 TX 队列，并交给 igb_xmit_frame_ring() 继续完成 DMA 映射、TX 描述符填充和通知硬件发送。

igb_xmit_frame_ring

igb_xmit_frame_ring() 是 igb 驱动真正处理单个发送包的核心函数。它首先根据 skb 的线性区和分片数量计算本次发送大概需要多少个 TX 描述符，并通过 igb_maybe_stop_tx() 判断发送环是否还有足够空间；如果空间不足，就暂时停止发送队列并返回 NETDEV_TX_BUSY。如果空间足够，驱动会在 tx_buffer_info[next_to_use] 中记录这个 skb 的软件发送上下文，包括长度、协议类型、TSO/VLAN/PTP 时间戳等信息。随后根据 skb 的特性分别处理硬件时间戳、VLAN tag、TSO 分段卸载或者普通校验和卸载，最后调用 igb_tx_map() 把 skb 的数据区 DMA 映射成 TX 描述符，并更新 tail 通知硬件开始发送

IGB 网卡启动流程-igb_open（五）

发表于 2026-06-23 更新于 2026-06-28 分类于 IGB网卡驱动 Valine：

igb_open

在前面分析 igb_probe 的时候，已经知道 igb 驱动是如何被 PCI 子系统匹配到、如何分配并初始化 net_device，最后通过 register_netdev() 把网卡注册到内核网络子系统中的。但是probe 完成并不代表网卡已经真正开始收发数据。此时驱动只是把设备“注册好了”，用户态能够看到这个网卡设备，例如通过 ip link 或 ifconfig -a 查看它。只有当用户执行 ip link set ethX up，或者系统网络服务拉起该网卡时，内核才会调用驱动提供的 ndo_open 回调，也就是 igb 驱动中的 igb_open()。它会完成真正运行前的一系列准备工作：申请和初始化描述符资源、配置硬件队列、注册中断、使能NAPI、使能发送接收路径，并最终让网卡具备收发报文的能力。igb_open具体代码如下所示：

__igb_open() 是 首先检查当前网卡是否处于测试状态，避免 ethtool 自检等场景下被错误打开；随后调用 netif_carrier_off() 告诉协议栈当前链路还不可用。接着调用igb_setup_all_tx_resources和igb_setup_all_rx_resources依次分配 TX/RX 描述符资源，拉起 PHY 链路，调用 igb_configure() 完成网卡硬件配置，包括收发路径、RX buffer、RSS、过滤规则等内容，再调用 igb_request_irq() 注册中断处理函数。之后，驱动把实际可用的 TX/RX 队列数量通知给网络协议栈，清除 __IGB_DOWN 状态位，开启 NAPI，清除历史挂起中断，并真正使能网卡中断。最后，驱动启动发送队列，并调度 watchdog 任务去异步检测真实链路状态；只有后续 watchdog 确认 link up 后，才会通过 netif_carrier_on() 通知协议栈链路可用。

在网络设备注册过程中，最终会进入 register_netdevice()，把驱动初始化好的 net_device 正式加入内核网络子系统。除了注册名字、地址、状态等基础信息外，内核还需要整理这个设备支持和当前启用的各种 offload 能力，也就是 dev->features。

这些 feature 不能只看驱动设置，因为它们之间存在依赖关系，比如 TSO 需要 SG 和 checksum offload；同时它们还可能受到 bridge、bond等设备关系的影响。所以注册过程中会调用__netdev_update_features()，修正并应用最终的 feature 集合。具体代码如下所示：

_netdev_update_features() 的作用是重新计算并更新设备最终生效的 dev->features。它先通过 netdev_get_wanted_features() 得到当前希望开启的 feature，然后依次经过驱动的 ndo_fix_features() 和内核通用的 netdev_fix_features() 修正，处理 feature 之间的依赖关系，比如 TSO 依赖 SG 和 checksum offload。接着它会根据 upper 设备的状态继续修正，保证上层设备关闭的某些能力，下层设备不能单独开启。如果最终 features 和当前 dev->features 不同，就调用驱动的 ndo_set_features() 重新配置硬件；如果没变，也会继续同步 lower 设备。最后针对 UDP tunnel offload、VLAN C-tag/S-tag filter 这些特殊 feature 做额外同步，并把结果写回 dev->features。

igb_watchdog_task() 是 igb 驱动中的周期性状态维护任务，主要负责链路状态检测、carrier 状态同步以及异常恢复处理。函数开始会通过 igb_has_link() 检查当前物理链路状态，并根据 等场景对 link 结果做修正。如果检测到链路处于 up 状态，并且内核当前认为 carrier 仍然是 down，说明链路刚刚恢复，此时驱动会读取网卡速率和双工模式，打印 link up 信息，检查 EEE、SmartSpeed 降速、温度降速等状态，然后调用 netif_carrier_on() 通知网络栈该设备已经具备发送能力。相反，如果检测到链路 down，而内核当前 carrier 仍然是 up，说明链路刚刚断开，驱动会清空速率和双工信息，打印 link down 日志，并调用 netif_carrier_off() 通知网络栈停止继续向该设备下发数据，同时根据需要进行介质切换或调度复位任务。链路状态处理完成后，watchdog 还会更新设备统计信息，检查 Tx ring 是否存在未完成描述符，设置 Tx hang 检测标志，主动触发一次软件中断以推动 Rx ring 清理，最后，如果设备没有处于 down 状态，则重新设置 watchdog_timer。

igb 网卡初始化-igb_probe（一）

发表于 2026-06-12 更新于 2026-06-28 分类于 IGB网卡驱动 Valine：

在 Linux 中，网卡驱动并不是主动去“扫描”硬件设备，而是遵循内核的设备驱动模型。以 igb 驱动为例，系统启动时 PCI 子系统会先枚举机器上的 PCI 设备，读取每个设备配置空间中的 vendor id、device id 等信息；而当 igb 驱动被内置进内核，或者通过 insmod / modprobe 加载时，它会通过 pci_register_driver() 把自己的 pci_driver 注册到 PCI 子系统中。随后，driver core 会将已经枚举到的 PCI 设备和 igb 驱动的 id_table 进行匹配。如果某个 Intel 网卡设备匹配成功，内核就会调用 igb 驱动提供的 probe 回调函数，也就是 igb_probe()。因此，igb_probe() 可以看作 igb 驱动接管一块具体网卡的入口函数，后续 PCI 设备使能、BAR 映射、net_device 分配、MAC/PHY/NVM 初始化、中断和队列初始化，以及最终的 register_netdev()，都是从这里展开的。

igb_probe() 是 igb 驱动接管一块 Intel PCI 网卡的入口函数。它首先会做 PCI 设备层面的初始化：检查当前设备不能是 VF，调用 pci_enable_device_mem() 使能 PCI memory 设备，设置 DMA mask，申请 PCI BAR 资源，并通过 pci_iomap() 把 BAR0 映射到内核虚拟地址。到这一步，驱动才具备访问网卡 MMIO 寄存器的能力。

接着，驱动会分配并初始化 Linux 网络设备对象。alloc_etherdev_mq() 会分配 net_device，同时预留 igb_adapter 作为驱动私有数据。随后驱动把 pdev、netdev、adapter、e1000_hw 这些结构关联起来，并设置 netdev_ops、ethtool_ops、watchdog 超时时间、BAR 地址范围等信息。这里完成的是“Linux 网络设备对象”和“PCI 硬件设备对象”之间的绑定。

然后，igb_probe() 会根据 PCI device id 找到对应的硬件信息表 e1000_info，拷贝 MAC、PHY、NVM 的操作函数，并调用 get_invariants() 获取具体芯片的不变量信息，比如 MAC 类型、PHY 类型、介质类型、硬件能力等。之后调用 igb_sw_init() 初始化驱动软件状态，包括队列数量、RSS、中断能力、ring 参数、feature 标志等，为后续收发队列和中断初始化做准备。

再往后，驱动开始配置 net_device 支持的各种硬件能力，例如 SG、TSO、TSO6、RX checksum、HW checksum、SCTP CRC、UDP GSO、VLAN offload、GSO partial、NTUPLE、XDP redirect 等。同时设置 MTU 范围、VLAN/MPLS/封装场景下的 offload 能力。这一部分决定了用户态通过 ethtool -k 能看到哪些网卡卸载能力，也决定了协议栈后续能不能把某些工作交给硬件处理。

在真正注册网卡之前，驱动会先 reset 一次控制器，让硬件处于一个已知状态，然后校验 NVM/EEPROM，读取 MAC 地址并设置到 netdev->dev_addr，同时把默认 MAC 地址写入硬件过滤表。随后初始化固件版本、packet buffer、watchdog 定时器、PHY 信息定时器、reset work 和 watchdog work，并设置链路自协商、流控、MDI/MDI-X、WoL 网络唤醒等默认配置。

完成这些配置后，驱动调用 igb_reset() 根据前面设置重新复位并初始化硬件，使 packet buffer、流控、EEE、VLAN、PTP、MAC filter 等配置真正写入硬件。接着初始化 I2C 接口，通知管理固件当前硬件已经由操作系统驱动接管，然后调用 register_netdev() 把这个网卡正式注册到 Linux 网络子系统。注册成功后，用户态就能通过 ip link、ethtool 等工具看到这个网卡。

最后，驱动会先调用 netif_carrier_off()，表示虽然网卡已经注册成功，但当前还不能直接认为物理链路是 up 的，真正的 link 状态要等后续 open 和 watchdog 检测后再通过 netif_carrier_on() 更新。后面还会继续初始化 DCA、温度传感器、Media Autosense、PTP、EEE 等附加功能，并打印 PCIe 速率、宽度、中断模式、队列数量、PBA 编号等信息。整体来看，igb_probe() 的作用就是把一块 PCI 网卡从“被系统枚举到的硬件设备”，一步步初始化成 Linux 网络子系统里可见、可配置、后续可打开收发包的 net_device。
