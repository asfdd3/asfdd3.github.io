引言

在 Linux 内核网络协议栈中，流量控制（Traffic Control，TC）位于发送路径的关键位置，负责在数据包进入网卡之前完成排队与调度。内核在网络设备初始化和激活阶段，就已经为每个设备构建了一套默认的队列体系，而对于多队列网卡，则会建立分层结构，将流量分发到不同的发送队列，并为每个队列配置独立的排队策略。与此同时，这些队列规则会被绑定到设备的发送路径上，使得所有数据包在发送前都会经过统一的调度入口。后续用户后可以通过 tc 命令对队列规则的配置，替换和重组这套已经存在的调度结构。

路由相关数据结构

发表于 2026-03-24 更新于 2026-03-23 分类于 网络协议栈源码学习 Valine：

引言

在 Linux 6.6 内核中，路由子系统不再依赖早期的缓存机制，而是以 fib_trie 前缀树为核心，实现高效的路由查找。整个子系统围绕“从策略到前缀匹配，再到下一跳决策”这一主线展开：入口由策略路由规则（fib_rule）决定查询路径，路由表（fib_table）承载具体数据，内部通过压缩前缀树（fib_trie / key_vector）完成最长前缀匹配；同一前缀下的多条路由以 fib_alias 链表组织，并通过 fib_info 进行去重和共享，最终由 fib_nh 描述下一跳信息，生成 rtable 作为查找结果供协议栈使用。相关的关键数据结构如下：

fib_table

fib_table 从逻辑上看，对应用户空间中的一张路由表，但在实现上，它本质是“路由表入口 + 前缀树的封装”，具体结构如下所示：

key_vector

key_vector 是前缀树中的核心节点结构，用于表示一个前缀匹配状态。在查找过程中，内核从根节点开始，根据 pos 和 bits 从目标 IP 地址中取出相应的位作为索引。当节点为中间节点时，通过 tnode 指向下一层子节点；当节点为叶子节点时，通过 leaf 挂载 fib_alias 链表，从而关联具体的路由信息。具体字段如下所示：

fib_alias

fib_alias 表示某一前缀节点下的具体路由项，在查找过程中，内核通过 key_vector 定位到匹配的前缀节点后，会在其 leaf 链表中遍历多个 fib_alias，每个 fib_alias 对应一条候选路由，并通过 fib_info 关联具体的下一跳信息。因此，fib_alias 的作用是承载同一前缀下的多条路由。
