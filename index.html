ARP初始化（一）

发表于 2026-05-06 更新于 2026-05-08 分类于 网络协议栈源码学习 Valine：

引言

邻居子系统是网络协议栈中负责下一跳邻居管理的通用框架，它主要解决的问题是当内核已经知道数据包该从哪个网卡、发给哪个 IP 后，如何进一步找到对应的二层地址（MAC），以及如何维护这些邻居的可达状态。为此，内核实现了一整套通用机制，包括邻居表管理、状态机、定时器、探测重传、缓存队列以及老化回收等功能。而 ARP 则是这个通用框架在 IPv4 下的具体实现，负责通过 ARP Request/Reply 报文完成 IPv4 地址到 MAC 地址的解析。

ARP 报文格式

字段含义

字段 含义 常见值
Hardware Type 链路层类型 1（Ethernet）
Protocol Type 上层协议类型 0x0800（IPv4）
HLen MAC 地址长度 6
PLen IP 地址长度 4
Opcode ARP 操作类型 1=Request，2=Reply
Sender MAC 发送方 MAC 本机 MAC
Sender IP 发送方 IP 本机 IP
Target MAC 目标 MAC Request 中通常为全0
Target IP 目标 IP 要解析的 IP

初始化

arp_init() 是 Linux 内核 ARP 模块的初始化入口，它主要完成 ARP 与整个网络协议栈的注册和关联。首先通过 neigh_table_init() 将 arp_tbl 注册到通用邻居子系统中。随后通过 dev_add_pack() 注册 ARP 报文接收入口 arp_packet_type，这样网卡收到以太网类型为 0x0806 的 ARP 报文就会进入 arp_rcv() 处理流程。接着 register_pernet_subsys() 会为每个网络命名空间创建对应的 ARP 相关数据结构和 proc/sysctl 接口，使不同 namespace 之间的 ARP 配置和邻居表能够互相隔离。neigh_sysctl_register() 则用于注册邻居子系统相关的 sysctl 参数，方便用户动态调整邻居行为。最后通过 register_netdevice_notifier() 注册网卡事件通知链，这样当网卡 up/down、地址变化或设备注销时，ARP 模块也能同步更新对应的邻居状态和 ARP 表项。

GRO（一）

发表于 2026-04-22 更新于 2026-04-21 分类于 网络协议栈源码学习 Valine：

引言

在内核协议栈中，性能是一个核心问题：如何减少每个数据包带来的处理开销。相比发送路径中的 TSO/GSO，接收路径同样存在类似的问题——网卡不断上报大量小包，而协议栈却需要逐个处理，无疑增加了CPU的使用率。当大量数据包涌入系统时，即使带宽没有打满，CPU 也可能因为频繁进入协议栈而被拖垮。为优化上述问题，协议栈引入了GRO（Generic Receive Offload），把多个连续的包合并成一个大包，减少协议栈处理次数。

GRO的入口是napi_gro_receive（网卡驱动上送数据包的接口），在推送数据包给协议栈之前，会在当前一个轮次的napi中尝试合并数据包，具体代码如下所示：

sch_direct_xmit() 的核心流程是，先释放 qdisc 的 root_lock（避免发包长时间占锁），在锁外调用validate_xmit_skb_list对 skb 做发送前校验（其实就是看网卡是否支持硬件卸载，不支持就交给软件来做），之后加 HARD_TX_LOCK，并判断发送队列是否被 stop/frozen 如果没有则调用 dev_hard_start_xmit 尝试发包，发送完成后检查返回值，如果驱动返回 NETDEV_TX_BUSY ，就把 skb 重新入队到 qdisc。
