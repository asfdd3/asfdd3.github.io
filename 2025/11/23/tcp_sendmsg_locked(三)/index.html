之后调用skb_copy_to_page_nocache将用户态数据拷贝到内核，并根据是否可以来更新页的相关管理字段。

如果是第一次创建数据包，则还会清除这个数据包的psh标志位，并更新发送缓冲区的序列号和skb的end_seq,接下来判断用户数据是否已经全部拷贝完，如果完成了则直跳出循环，否则判断当前skb是否还能继续填充数据，如果可以，则继续循环拷贝数据，直到数据包拷贝完成或者当前数据包已经填充size_goal大小的数据包才会结束。

这里分为两种情况，一是如果当前数据包已经填充满了(达到了size_goal),则判断是否需要forced_push（如果当前写入的数据量已经超过发送窗口的一半，那就必须强制 push。如果需要则标记push标志，并调用__tcp_push_pending_frames发送队列中的数据包。

而是如果当前数据包是发送队列的头部，则只发送这一个数数据包（这个概率相对较小吧）。

如果是循环内部由于数据包已经全部拷贝完成跳出循环，则会调用tcp_push来发送数据包，具体代码如下所示：

tcp_push应该是是tcp_sendmsg_locked中被调用次数最多的函数，首先拿到队尾 skb，如果没有设置msgmore 或 达到了窗口大小的一半，就给数据包打上 PUSH 标记，

接着 TCP 会判断是否需要autocork(当前数据包小于GSO大小，系统启用了cork(默认开启)，有数据包在途中，有数据待发送)，如果满足上述条件则会设置TSQ标志位(数据包的析构函数会调用软中断发送数据包），如果有数据还在协议栈，延迟发送。否则调用

__tcp_push_pending_frames发送数据包。

总结：tcp_sendmsg_locked中存在三个函数完成进一步向下发包的工作，分别是__tcp_push_pending_frames，tcp_push_one和tcp_push。三者主要区别和调用的场景如下所示：

函数 所在层级 主要作用 在 tcp_sendmsg_locked() 里的场景
tcp_push() 最高层 做 MSG_MORE / Nagle / TSQ / autocork 的决策，然后统一 push 一次 send()/write() 结束，或者内存不足但已拷贝一部分数据时，整体考虑是否发送
tcp_push_one() 中间层 只发送 tcp_send_head() 这一个 skb，启动流水线但不 flush 整个队列 在循环中，当前 skb 已写满 size_goal、还没到 forced_push 阈值，但又是队列头，小心地发一个包
__tcp_push_pending_frames() 最底层 不再做策略判断，直接根据 cwnd/窗口，把 pending 的 skb 尽量发给 IP 层 在循环中判断到 forced_push(tp)，说明窗口已经占得比较满了，需要强制把队列里的数据尽量冲出去
