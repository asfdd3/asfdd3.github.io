igb_watchdog

完成网卡基本资源初始化之后，驱动还需要建立一些异步处理机制，像链路状态变化、发送队列是否hang住、是否需要复位，这些都需要看门狗机制进行检查。igb 驱动这里就初始化了watchdog_timer 定时触发 igb_watchdog，再由它调度真正干活的 igb_watchdog_task。具体代码如下所示：

igb_watchdog_task() 是 igb 驱动中的周期性状态维护任务，主要负责链路状态检测、carrier 状态同步以及异常恢复处理。函数开始会通过 igb_has_link() 检查当前物理链路状态，并根据 等场景对 link 结果做修正。如果检测到链路处于 up 状态，并且内核当前认为 carrier 仍然是 down，说明链路刚刚恢复，此时驱动会读取网卡速率和双工模式，打印 link up 信息，检查 EEE、SmartSpeed 降速、温度降速等状态，然后调用 netif_carrier_on() 通知网络栈该设备已经具备发送能力。相反，如果检测到链路 down，而内核当前 carrier 仍然是 up，说明链路刚刚断开，驱动会清空速率和双工信息，打印 link down 日志，并调用 netif_carrier_off() 通知网络栈停止继续向该设备下发数据，同时根据需要进行介质切换或调度复位任务。链路状态处理完成后，watchdog 还会更新设备统计信息，检查 Tx ring 是否存在未完成描述符，设置 Tx hang 检测标志，主动触发一次软件中断以推动 Rx ring 清理，最后，如果设备没有处于 down 状态，则重新设置 watchdog_timer。

register_netdevice

完成上述一些列流程后，最后需要调用 register_netdevice() 将之前申请的网络设备正式注册到内核网络子系统中，只有完成这一步，内核才能通过统一的网络设备框架管理它，用户态也才能看到对应的网卡接口。

register_netdevice() 开始会先做一些基本检查，比如确认设备还没有被注册过、设备所属的网络命名空间有效， ethtool 操作是否合法。随后检查并分配合法的网卡名称，接下来，内核会检查驱动声明的 VLAN 硬件卸载能力是否和对应回调匹配，然后为设备分配 ifindex，并整理 features、hw_features、wanted_features 等特性字段：其中 features 表示当前启用的能力，hw_features 表示硬件可支持的能力，wanted_features 则表示用户期望开启且硬件支持的能力集合。完成这些准备后，内核会发送 NETDEV_POST_INIT 通知，并通过 netdev_register_kobject() 在 sysfs 下创建 /sys/class/net/<dev> 相关节点，使用户态可以查看和配置该网卡。随后调用 __netdev_update_features() 更新设备特性，初始化 linkwatch，并通过 dev_init_scheduler() 为设备初始化 qdisc 和网络设备层的发送 watchdog。最后，list_netdevice() 会把设备加入当前网络命名空间的设备链表、按名称的 hash 表和按 ifindex 的 hash 表，这样协议栈后续才能通过名字、索引或遍历找到该设备。
