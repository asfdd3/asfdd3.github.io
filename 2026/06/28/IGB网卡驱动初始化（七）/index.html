igb_alloc_rx_buffers

在 RX ring 初始化或 NAPI 清理完已使用的接收描述符后，驱动需要把这些空出来的描述符重新挂上可 DMA 的接收缓冲区，具体代码如下所示：

igb_alloc_rx_buffers() 先从 next_to_use 指向的位置开始，找到对应的 RX 描述符 rx_desc ，然后循环补充 cleaned_count 个描述符。每补一个位置，就先通过 igb_alloc_mapped_page() 确保这个 bi 里有可用的 page，接着把这段 buffer 同步给设备使用；然后把 bi->dma + bi->page_offset 写入 RX 描述符的 pkt_addr，告诉网卡后续可以把收到的数据 DMA 到这里。写完一个描述符后，软件指针往后推进，循环结束后，如果确实补了新的描述符，就更新 next_to_use 和 next_to_alloc，再通过 dma_wmb() 保证前面对描述符的写入先完成，最后 writel(i, rx_ring->tail) 把新的 tail 写给硬件。

上述igb_alloc_mapped_page具体实现如下所示：

igb_alloc_mapped_page() 的作用就是确保当前 bi 有一块网卡可 DMA 写入的 page。如果 bi->page 已经存在，说明这个 buffer 可以复用，直接返回；如果没有，就申请新的 page，普通 MTU 下一般是 order 0，也就是一个 4K 页。申请成功后，通过 dma_map_page_attrs() 做 DMA 映射，方向是 DMA_FROM_DEVICE，表示后续数据是网卡写到内存。映射失败就释放 page 并记录失败次数；映射成功后，把 dma、page 和 page_offset 保存到 bi 中。最后通过 page_ref_add(page, USHRT_MAX - 1) 和 pagecnt_bias = USHRT_MAX 一次性预加很多引用计数，让驱动后续用本地 pagecnt_bias 记账，减少频繁操作 page 引用计数的开销，这是 igb 复用 page 的关键。

igb_request_irq() 是 igb 申请（注册）中断的统一入口，按 MSI-X → MSI → legacy INTx 的顺序尝试。优先走 MSI-X 时，会调用 igb_request_msix()失败降级到 MSI；MSI 再失败则使用传统共享中断 igb_intr()。

igb_request_msix

igb_request_msix具体实现如下所示：

igb_request_msix() 的作用是在 MSI-X 模式下为 igb 注册各个 IRQ handler，并完成硬件 MSI-X 映射配置。它先使用 msix_entries[0] 申请一个中断，handler 是 igb_msix_other()，用于处理链路变化、reset、mailbox、时间戳等非收发队列事件；然后从 vector = 1 开始，遍历每个 q_vector，给它设置对应的 EITR 中断节流寄存器地址，生成一个用于 /proc/interrupts 显示的中断名字，再调用 request_irq() 把该 MSI-X vector 绑定到 igb_msix_ring()，用于 RX/TX 队列中断并调度 NAPI。所有队列中断申请成功后，调用 igb_configure_msix() 把 other 事件和各个 RX/TX 队列映射到对应的 MSI-X vector 上。

igb_msix_other具体实现如下所示:

igb_msix_other() 是 MSI-X 的非队列中断处理函数，不负责收发包。它读取 E1000_ICR 判断中断原因：如果是 reset，就调度 reset_task；如果是 DMA 异常，就统计并检查是否有 VF spoof；如果是 mailbox，就处理 VF 消息；如果是链路变化，就标记需要重新检测 link，并马上触发 watchdog；如果是 PTP 时间戳事件，就处理时间戳。最后重新使能 other 中断。

igb_msix_ring具体实现如下所示：

igb_msix_ring() 是 MSI-X 队列中断处理函数，负责 RX/TX 队列中断。它拿到当前中断对应的 q_vector 后，先调用 igb_write_itr() 更新这个队列 vector 的中断节流时间，然后调用 napi_schedule() 把对应的 NAPI 加入软中断轮询队列。
