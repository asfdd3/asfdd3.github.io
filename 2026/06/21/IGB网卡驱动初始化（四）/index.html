IGB网卡驱动初始化（四）

发表于 2026-06-21 分类于 IGB网卡驱动 阅读次数： Valine：

在网络设备注册过程中，最终会进入 register_netdevice()，把驱动初始化好的 net_device 正式加入内核网络子系统。除了注册名字、地址、状态等基础信息外，内核还需要整理这个设备支持和当前启用的各种 offload 能力，也就是 dev->features。

这些 feature 不能只看驱动设置，因为它们之间存在依赖关系，比如 TSO 需要 SG 和 checksum offload；同时它们还可能受到 bridge、bond等设备关系的影响。所以注册过程中会调用__netdev_update_features()，修正并应用最终的 feature 集合。具体代码如下所示：

_netdev_update_features() 的作用是重新计算并更新设备最终生效的 dev->features。它先通过 netdev_get_wanted_features() 得到当前希望开启的 feature，然后依次经过驱动的 ndo_fix_features() 和内核通用的 netdev_fix_features() 修正，处理 feature 之间的依赖关系，比如 TSO 依赖 SG 和 checksum offload。接着它会根据 upper 设备的状态继续修正，保证上层设备关闭的某些能力，下层设备不能单独开启。如果最终 features 和当前 dev->features 不同，就调用驱动的 ndo_set_features() 重新配置硬件；如果没变，也会继续同步 lower 设备。最后针对 UDP tunnel offload、VLAN C-tag/S-tag filter 这些特殊 feature 做额外同步，并把结果写回 dev->features。

default_operstate() 先根据当前设备状态算一个结果：如果设备在 testing，就返回 TESTING；如果没有 carrier，就说明链路没起来，再看它有没有下层设备，如果是普通网卡自己没链路，就返回 DOWN，如果是 VLAN、bond 这类上层设备，并且下层设备也没链路，就返回 LOWERLAYERDOWN；如果链路已经起来了，但还处于 dormant 状态，比如 802.1X 认证没完成，就返回 DORMANT；否则就返回 UP。

rfc2863_policy() 拿到这个结果后，如果变了，就更新它。另外它还会看一下 dev->link_mode，这个一般是用户态设置的，最后写回 dev->operstate。

dev_watchdog() 就是网络设备的发送超时看门狗。它会在设备已经打开、链路也是 up、qdisc 正常的情况下，遍历所有 TX 队列，检查netif_xmit_stopped(txq) 为真(驱动设置的)，并且距离上次发包时间 txq->trans_start 已经超过 dev->watchdog_timeo。如果发现超时，就调用驱动的 ndo_tx_timeout() 回调（本质就是复位网卡）。最后它会重新挂上定时器，继续下一轮检查。

netif_carrier_on

netif_carrier_on() 是驱动在检测到物理链路已经起来时调用的接口，用于通知协议栈网卡已经link up，只有调用这个接口之后，协议栈才会把包交给网卡驱动，否者在down的情况下，数据包都被tc的noop qdisc给丢弃了。netif_carrier_on具体实现如下所示：

__linkwatch_run_queue() 是 linkwatch 真正消费事件队列的地方。前面 netif_carrier_on/off() 只是把发生链路变化的设备挂到全局 lweventlist，这个函数会把队列里的设备取出来批量处理。处理时会跳过当前不适合处理的设备，真正需要处理的设备则调用 linkwatch_do_dev(dev)，由它去更新 operstate、激活或停用发送队列，并通知用户态。最后如果队列里还有没处理完的设备，就继续调度下一次 linkwatch work。
