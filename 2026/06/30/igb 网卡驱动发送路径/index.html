igb 网卡驱动发送路径

发表于 2026-06-30 更新于 2026-06-29 分类于 IGB网卡驱动 阅读次数： Valine：

前面已经分析了 igb 网卡驱动的初始化流程，包括设备探测、资源申请、队列初始化以及 open 阶段对硬件寄存器的配置。网卡完成初始化之后，真正的数据发送入口就是 ndo_start_xmit 回调。在 igb 驱动中，这个回调对应的是 igb_xmit_frame() 函数。协议栈在需要发送数据包时，会把构造好的 skb 交给该函数，由驱动完成 TX 队列选择、描述符填充、DMA 映射以及通知硬件发送等工作。本文从 igb_xmit_frame 这一入口开始，梳理 igb 网卡驱动的发送路径。

igb_xmit_frame

这段代码就是 igb 驱动的发包总入口：先从 netdev 找到网卡私有结构 adapter，然后确保待发送的 skb 至少有 17 字节，满足硬件最小发送长度要求，最后根据 skb 选择对应的 TX 队列，并交给 igb_xmit_frame_ring() 继续完成 DMA 映射、TX 描述符填充和通知硬件发送。

igb_xmit_frame_ring

igb_xmit_frame_ring() 是 igb 驱动真正处理单个发送包的核心函数。它首先根据 skb 的线性区和分片数量计算本次发送大概需要多少个 TX 描述符，并通过 igb_maybe_stop_tx() 判断发送环是否还有足够空间；如果空间不足，就暂时停止发送队列并返回 NETDEV_TX_BUSY。如果空间足够，驱动会在 tx_buffer_info[next_to_use] 中记录这个 skb 的软件发送上下文，包括长度、协议类型、TSO/VLAN/PTP 时间戳等信息。随后根据 skb 的特性分别处理硬件时间戳、VLAN tag、TSO 分段卸载或者普通校验和卸载，最后调用 igb_tx_map() 把 skb 的数据区 DMA 映射成 TX 描述符，并更新 tail 通知硬件开始发送

igb_tso() 首先判断当前 skb 是不是 CHECKSUM_PARTIAL 或者是不是 GSO 包，就说明不需要 TSO，直接返回 0。对于真正的 TSO 包，它会先确保 skb 头部可写，然后根据 IPv4/IPv6、TCP/UDP 类型修改协议头中的长度和校验和字段，把原始大包调整成硬件可以分段处理的格式。接着记录分段数量、修正发送字节统计，并根据 MAC 头长度、IP 头长度、L4 头长度、MSS 等信息构造上下文描述符。

igb_tx_csum() 用来处理非 TSO 场景下的硬件校验和卸载。如果 skb 不需要 CHECKSUM_PARTIAL，并且也没有 VLAN 或 launchtime 这类额外信息，就直接返回。否则它会根据 csum_offset 判断 TCP、UDP 或 SCTP 校验和类型，设置对应的硬件标志；如果硬件无法处理，就退回软件计算。最后通过上下文描述符把 MAC 头长度、IP 头长度、VLAN 和校验和信息交给硬件。

igb_tx_map

回到igb_xmit_frame_ring中，igb_tx_map() 是 igb 驱动发送路径中真正把 skb 交给硬件前的最后一步：它把 skb 的线性区和分页片段做 DMA 映射，填充 TX 数据描述符，最后更新 TDT tail 寄存器通知网卡开始 DMA 取数据并发送。具体代码如下所示：

igb_tx_map() 先对 skb 的线性区做 DMA 映射，然后依次处理 skb 的分页片段，把每一段数据的 DMA 地址和长度写入 TX 描述符；如果单段数据超过硬件单个描述符支持的最大长度，还会拆成多个描述符。最后一个描述符会设置 EOP 和 RS 等标志，表示一个完整数据包结束并需要硬件回写状态。所有描述符填好后，驱动更新 next_to_use，必要时停止发送队列，并在合适时机写 tail 寄存器通知网卡开始取描述符发送。
