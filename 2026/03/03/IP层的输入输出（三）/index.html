传输层到 IP 层的输出接口

发表于 2026-03-03 分类于 网络协议栈源码学习 阅读次数： Valine：

在内核网络协议栈中，当传输层封装好对应的数据包后，会调用IP层的发包接口完成IP头部的构造，路由查找等处理流程，以TCP为例，TCP 在发送数据时通常通过 ip_queue_xmit() 作为封装入口，将构造好的 skb 交给 IPv4 子系统。

ip_select_ident_segs首先判是否为本地 sk且已经有目的地址，有的话就用该sk私有的 inet_id 计数器生成 IPID，并考虑GSO的情况，保证一个大包拆成多个段时 ID 连续。若没有 sk 或不走这条路径，则判断 DF 位：如果设置了不允许分片且不忽略 DF，说明这个包绝不会分片，IPID 直接置 0 即可；否则就调用 __ip_select_ident()，根据 IP 和协议做 hash，定位到一个桶，并引入了时间戳和随机扰动：如果这个桶一段时间没用，就随机跳过一段 ID（delta），这样即使观察到 IPID 的变化，也难以推断中间真实发送了多少包，从而提升安全性。

__ip_append_data

__ip_append_data是一个相当复杂的函数，可以理解为UDP和RAW套接字的输出接口，核心思想就是把用户的传下来的数据包封装成多个skb（MTU以下），为后续分片做准备工作，注意__ip_append_data没有进一步向下发送数据包，仅仅是封装数据包到发送队列中，通常由外层函数进一步处理后完成发送工作，具体代码如下所示：

上述 __ip_append_data核心思想为把用户要发送的一大段数据，按 MTU 的大小切成几段，装进多个 skb 里，并把这些 skb 依次放进 queue 等待后面真正发出去。它先根据路由设备算出二层头预留长度、IP 头长度，以及每个分片最多能装多少数据（还要满足 IP 分片 8 字节对齐），同时检查总长度别超过 IPv4 的上限；然后进入循环：如果队列尾部已有 skb 还能塞数据，就继续把数据拷进去；如果塞不下了，就新分配一个 skb（预留二层空间、设置 network/transport header 的偏移、初始化 checksum 等元数据），把本段数据拷进去并入队。过程中还会处理第一个分片带传输层头、后续分片不再带的情况，同时维护 cork->length 和 sk_wmem_alloc 等发送内存记账；如果中途分配失败或拷贝失败，就回滚长度、做丢包统计并返回错误。核心思想就是：把 sendmsg 给的用户数据整理成一串符合 MTU/分片规则的 skb 链。
