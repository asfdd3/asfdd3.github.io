路由相关数据结构

发表于 2026-03-24 更新于 2026-03-23 分类于 网络协议栈源码学习 阅读次数： Valine：

引言

在 Linux 6.6 内核中，路由子系统不再依赖早期的缓存机制，而是以 fib_trie 前缀树为核心，实现高效的路由查找。整个子系统围绕“从策略到前缀匹配，再到下一跳决策”这一主线展开：入口由策略路由规则（fib_rule）决定查询路径，路由表（fib_table）承载具体数据，内部通过压缩前缀树（fib_trie / key_vector）完成最长前缀匹配；同一前缀下的多条路由以 fib_alias 链表组织，并通过 fib_info 进行去重和共享，最终由 fib_nh 描述下一跳信息，生成 rtable 作为查找结果供协议栈使用。相关的关键数据结构如下：

fib_table

fib_table 从逻辑上看，对应用户空间中的一张路由表，但在实现上，它本质是“路由表入口 + 前缀树的封装”，具体结构如下所示：

key_vector

key_vector 是前缀树中的核心节点结构，用于表示一个前缀匹配状态。在查找过程中，内核从根节点开始，根据 pos 和 bits 从目标 IP 地址中取出相应的位作为索引。当节点为中间节点时，通过 tnode 指向下一层子节点；当节点为叶子节点时，通过 leaf 挂载 fib_alias 链表，从而关联具体的路由信息。具体字段如下所示：

fib_alias

fib_alias 表示某一前缀节点下的具体路由项，在查找过程中，内核通过 key_vector 定位到匹配的前缀节点后，会在其 leaf 链表中遍历多个 fib_alias，每个 fib_alias 对应一条候选路由，并通过 fib_info 关联具体的下一跳信息。因此，fib_alias 的作用是承载同一前缀下的多条路由。

fib_info

fib_info 是 Linux 路由子系统中用于描述具体转发行为的核心结构，主要包含下一跳、出接口、优先级、路由类型以及传输相关参数（如 MTU、RTT 等）。与基于前缀组织的 fib_alias 不同，fib_info 并不关心匹配哪个网段，而是定义数据包匹配后应该如何转发。同时使多个不同前缀的路由（fib_alias）共享同一个 fib_info。

fib_nh

fib_nh 是描述下一跳的核心结构，。它封装了转发所需的关键信息，包括输出设备、下一跳网关地址、源地址、作用域等。

具体字段如下所示：

fib_result

struct fib_result 是在进行路由查找过程中用于保存结果的结构，用于描述一次查表命中的前缀、对应的路由信息，以及最终选定的下一跳。它在 fib_table_lookup 中被填充，为后续构造 rtable提供支持。该结构生命周期较短，仅在查路由阶段存在。

具体字段如下所示：

dst_entry

dst_entry 是协议栈中用于描述数据包转发路径的核心抽象结构，它封装了输出设备、转发处理函数（input/output）。作为 IPv4 的 rtable 的 rt6_info 的基类，dst_entry 会被挂载到 skb->dst 上，在数据包的发送和接收过程中影响。

具体字段如下所示：

rtable

struct rtable 继承自 dst_entry，用于表示一次路由查找后的最终转发表项。它在dst_entry的基础上，补充了 IPv4 特有的信息，例如网关地址（rt_gw）以及路径 MTU等。rtable 会被挂载到 skb->dst 上，在数据包的发送和接收过程中直接参与转发决策。
