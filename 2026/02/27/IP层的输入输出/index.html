IP报文的输入

上述ip_rcv_core() 核心思想就是针对接收的ipv4报文进行一系列的合法性检查，先丢弃混杂模式下收到的非本机目标帧。随后把 IP 头拉到线性区，检查版本，IP 头校验以及总长度合法性；若报文被链路层 padding，则会裁剪报文，并设置 transport_header 让上层能直接定位传输层头。

ip_rcv_finish中主要工作就是调用ip_rcv_finish_core，并根据返回值决定是否需要丢弃数据包，如果不丢弃数据包则根据返回值决定上述本机还是转发。上述ip_rcv_finish_core主要工作就是查路由（**也可能通过早期分流的机制直接拿到路由信息）**具体代码如下所示：

这段 ip_rcv_finish_core() 的作用是找到数据包的路由信息，首先尝试通过 early demux提前找到对应的 socket，并可能顺便拿到路由，如果还没有路由信息，就调用路由查找函数确定这个包是本机处理还是需要转发。接着处理 IP 选项，并根据路由类型统计是否是广播/组播，如果出现二层是广播但三层是单播的异常包，就按配置决定是否丢弃。

tcp_v4_early_demux() 核心思想就是在还没查路由前，先看看这个 TCP 包是不是已经有对应的sk。首先把 TCP 头拉到线性区，然后根据四元组（源/目的 IP+端口）直接在 ehash 里查找已建立连接的sk。如果找到了，就把 skb->sk 关联到这个 sk，并尝试把该连接缓存的路由也直接挂到 skb 上。这样后面流程就不用再重复查sk和路由了，相当于提前加速收包路径。

如果上述查找路由结果为上送本机则调用ip_local_deliver回调，ip_local_deliver主要工作就是完成数据包的重组工作（如果需要），这里先不具体分析，之后则调用LOCAL_DELIVER钩子，如果放行，则会进一步调用ip_local_deliver_finish根据ipproto分发给具体的传输层协议，上述具体代码如下所示：

这段 ip_protocol_deliver_rcu主要逻辑就是根据 IP 头里的 protocol 字段分发给对应的传输层协议。流程是先让 raw socket（IP层原始套接字）获取这个包，然后在 inet_protos 协议表中查找对应处理函数（例如tcp_v4_rcv），在通过 IPsec检查后调用该协议的接收入口，如果处理函数返回负值则需要再次分发（特殊的协议？），若找不到对应协议且也没有 raw socket 接收，则发送 ICMP 协议不可达，否则认为已被 raw socket 消费并释放 skb。
