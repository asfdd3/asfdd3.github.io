GRO（一）

发表于 2026-04-22 更新于 2026-04-21 分类于 网络协议栈源码学习 阅读次数： Valine：

引言

在内核协议栈中，性能是一个核心问题：如何减少每个数据包带来的处理开销。相比发送路径中的 TSO/GSO，接收路径同样存在类似的问题——网卡不断上报大量小包，而协议栈却需要逐个处理，无疑增加了CPU的使用率。当大量数据包涌入系统时，即使带宽没有打满，CPU 也可能因为频繁进入协议栈而被拖垮。为优化上述问题，协议栈引入了GRO（Generic Receive Offload），把多个连续的包合并成一个大包，减少协议栈处理次数。

GRO的入口是napi_gro_receive（网卡驱动上送数据包的接口），在推送数据包给协议栈之前，会在当前一个轮次的napi中尝试合并数据包，具体代码如下所示：

dev_gro_receive 首先判断当前设备是否开启 GRO，如果没开就直接走 normal 路径上送；如果开启了，就先根据 skb 的 hash 找到当前 CPU 上对应的 GRO bucket，并通过 gro_list_prepare() 先把这个 skb 和桶里已有包做一轮是否同流的预标记，然后按协议类型找到对应的 gro_receive 回调（比如 inet_gro_receive、tcp_gro_receive）去真正尝试和桶里的旧 skb 合并；在这个过程中会初始化当前 skb 的 GRO 控制信息，比如 flush、is_atomic、count 和 checksum 状态，如果回调返回了 pp，说明某个旧 skb 已经聚合完成，需要从链表摘下并调用 napi_gro_complete() 上送；如果当前 skb 最终仍然属于同流并成功并入已有包，就直接结束；如果不是同流、但又不需要立刻 flush，就把当前 skb 作为这个 bucket 里的一个新 GRO 流挂到链表上，等待后续同流包继续合并；如果既不能合并又不适合继续缓存，就走 normal 路径直接上送协议栈。

inet_gro_receive 首先检查当前 skb 的 IPv4 头是否合法（注意：必须是标准 20 字节头、不能带 IP option、不能是分片包、头校验和必须正确），然后把当前包的 L4 协议号、IP ID/DF 相关状态和初始 flush 信息记到 NAPI_GRO_CB(skb) 里；接着遍历当前 bucket 链表中的旧 skb，只对前面 gro_list_prepare() 标成 same_flow 的包继续比较，进一步检查源/目的 IP、协议号、TTL、TOS、DF 等字段是否一致，并计算当前包与旧包的 IP ID 差值，供后面的 TCP层合并的时候进一步决定是否可以合并，最后pull 掉 IPv4 头，调用下层的 tcp4_gro_receive() 去做真正的传输层聚合判断。
