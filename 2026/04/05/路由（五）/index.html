路由查找与反向路径过滤

__fib_lookup() 只是简单的包裹了一下 fib_rules_lookup ，fib_rules_lookup首先按顺序遍历 rule 链表，对于每条规则，先看源地址、目的地址、TOS、输入输出接口等条件是否匹配，匹配后再根据 action 处理，例如NOP 就直接跳过，普通情况则调用 fib4_rule_action() 按这条 rule 的配置去查指定路由表（例如 local、main、default 或用户自定义表）。fib4_rule_action具体实现如下所示：

fib4_rule_action() 的主要逻辑为，如果 rule 的 action 是 FR_ACT_TO_TBL，就先从 rule 中取出要查的路由表 ID，再根据这个 ID 找到对应的 fib_table，最后调用 fib_table_lookup() 在这张表里做真正的路由查找，这里注意无论是否配置策略路由，fib_table_lookup都是查找路由的真正入口函数，可以说是核心中的核心，例如其实没有开启策略路由fib_lookup中就会调用fib_table_lookup直接查找main表或者local表。

fib_table_lookup 理解起来比较困难：-），核心思想是将目地IP作为 key，在压缩前缀树的结构中自顶向下遍历，通过 key 与节点前缀的比较（异或 + 位偏移）逐步选择最优分支，优先逼近最长匹配，当不满足前缀约束时，通过记录的父节点与索引信息进行回溯，继续在更短前缀空间中搜索候选路径。最终在叶子节点处遍历挂载的 fib_alias 链表，对每个候选项进行语义过滤（前缀长度匹配、DSCP、scope、状态有效性等），并根据路由类型确定最终的 fib_nh_common，命中之后将对应的前缀、前缀长度、路由类型、作用域、fib_info 以及下一跳信息填充到 fib_result 中。

fib_validate_source核心逻辑是如果没有开启反向路径过滤且没有复杂策略路由，就根据是否允许本地地址等配置放行或拒绝，但只要开启了 rp_filter就直接进入 __fib_validate_source()，通过查路由表做完整的反向路径检查，判断该源地址从当前接口接收是否合理。

__fib_validate_source具体实现如下所示：

__fib_validate_source核心思想就是把收到数据包的源地址 src 反过来做一次路由查询，检查按照本机路由视角，这个源地址是否应该从当前这个接口 dev 进来。 它先构造一个反向查路条件，然后调用 fib_lookup() 查路，如果查不到，没开 rp_filter 就放行，如果查到了，但结果不是正常单播路由，或者不是被允许的本地路由，就认为源地址本身不合法，如果查出来的路由确实会走当前入接口，就校验通过，如果不走当前接口，而当前接口又没配地址，就走兜底逻辑，如果开的是严格反向路径过滤，就直接判失败，如果是宽松模式，就再强制指定当前接口做一次查路，只要这个源地址至少还能从这个接口解释得通，就放行，否则拒绝。
