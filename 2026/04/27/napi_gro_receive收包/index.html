napi_gro_receive收包流程

发表于 2026-04-27 分类于 网络协议栈源码学习 阅读次数： Valine：

在 Linux 内核网络收包路径中，通常会在 NAPI 轮询中完成收包，然后通过 napi_gro_receive() 将 skb 继续上送。这个函数是驱动 RX 路径和内核协议栈之间的一个重要入口，它的核心作用是：接收驱动构造好的 skb，尝试进行 GRO 合并，减少后续协议栈需要处理的数据包数量；如果不能合并，则把数据包继续交给通用收包路径处理。

首先通过 skb_mark_napi_id(skb, napi) 把当前 NAPI 的 ID 记录到 skb 中，这个信息后面可能被 busy poll 等机制使用，接着调用 初始化 GRO 处理需要用到的偏移字段，最后调用 dev_gro_receive 尝试把当前 skb 和 NAPI 上已有的 GRO 包进行合并（前文已经分析过，这里不在分析），再把合并结果交给 napi_skb_finish() 收尾处理。

napi_skb_finish() 是 GRO 处理后的收尾函数。它根据 dev_gro_receive() 的返回值决定 skb 怎么处理，如果是 GRO_NORMAL，说明这个包需要正常上送协议栈，就调用 gro_normal_one()；如果是 GRO_MERGED_FREE，说明这个 skb 已经被合并到别的 GRO 包里了，当前 skb 可以释放；如果是 GRO_HELD、GRO_MERGED、GRO_CONSUMED，说明 skb 已经被 GRO 暂存、合并或消费掉了，这里不用再处理。

gro_normal_one() 的作用是把需要正常上送协议栈的 skb 先挂到当前 NAPI 的 rx_list 上，暂时攒一批再统一上送。这样做可以减少每来一个包就立刻进入协议栈的开销。如果数量达到 gro_normal_batch 阈值，就调用 gro_normal_list 把这一批 skb 一起送入协议栈。

gro_normal_list() 的作用是把前面攒在 napi->rx_list 里的 skb 批量上送到协议栈。如果 rx_count 为 0，说明当前没有需要上送的包，直接返回；否则调用 netif_receive_skb_list_internal，把整个 skb` 链表一次性交给协议栈处理

netif_receive_skb_list_internal() 的重点就是把 NAPI/GRO 攒好的 skb 链表批量送入协议栈。在真正上送前，它会先遍历 skb 链表，处理接收时间戳，把需要等待硬件时间戳的 skb 暂时过滤掉，**然后如果开启了 RPS/RFS，就根据 skb 的 hash 选择目标 CPU，把部分 skb 转交到其他 CPU 的 backlog 队列处理；**最后剩下的 skb 才会通过 __netif_receive_skb_list 进入真正的协议栈接收流程。__netif_receive_skb_list具体实现如下所示：

__netif_receive_skb_list() 的重点是把一批 skb 按是否属于 pfmemalloc 场景分成不同子链表，再分别送入核心接收流程。正常情况下，大多数 skb 都不是 pfmemalloc，所以会直接走最后的 __netif_receive_skb_list_core，只有在系统内存紧张、skb 来自紧急内存池时，才会把普通 skb 和 pfmemalloc skb 分开处理。

__netif_receive_skb_list_core具体代码如下所示

__netif_receive_skb_list_core() 的核心思想是：先逐个 skb 调用 __netif_receive_skb_core() 找到最终要交给哪个协议处理函数，然后把“协议处理者相同、原始入口设备相同”的 skb 再攒成一个小链表，最后批量投递。

__netif_receive_skb_core具体实现如下所示：

__netif_receive_skb_core()可以理解为 进入协议栈前的核心分发函数。它先记录原始收包设备 orig_dev，先经过 generic XDP，如果 XDP 不是 PASS 就直接结束。接着处理 VLAN：如果 VLAN 头还在报文里，就剥掉VLAN头并保存到 skb中；然后经过抓包点 ptype_all，所以 tcpdump 这类 AF_PACKET 可以在这里看到包；之后进入 tc ingress / netfilter ingress，可能放行、丢包或者重定向。再往后，如果 skb 带 VLAN tag，会调用 vlan_do_receive() 根据 VLAN ID 找对应的 VLAN 子接口，找到后修改 skb->dev 并重新走 another_round。如果设备注册了 rx_handler，比如 bridge、bond、OVS，也会在这里接管 skb。最后根据 skb->protocol 查找协议处理入口：普通情况查全局 ptype_base，比如 IPv4 对应 ip_rcv()；同时还会查具体设备上的 ptype_specific。如果找到处理函数，就通过 ppt_prev 返回给外层继续投递；如果没有找到，就释放 skb，并增加 drop统计。
