网卡TX队列选择

发表于 2026-04-14 更新于 2026-04-15 分类于 网络协议栈源码学习 阅读次数： Valine：

在 Linux 内核发包路径中，skb 在进入驱动之前必须先确定一个发送队列（TX queue），对于多队列网卡，这个选择通常在 dev_queue_xmit 过程中通过 netdev_core_pick_tx() 完成，可能基于flow hash以及 XPS 配置等因素计算得到，最终结果写入 skb->queue_mapping 并决定后续使用的 struct netdev_queue。

netdev_core_pick_tx

netdev_core_pick_tx在__dev_queue_xmit种被调用，核心思想是在发包时为 skb 选择一个 TX 队列并记录，如果网卡是多队列设备，优先调用驱动提供的 ndo_select_queue，否则走内核默认的 netdev_core_pick_tx()（通常基于 flow hash / XPS 等），然后对得到的队列索引，最终通过 skb_set_queue_mapping() 写入 skb->queue_mapping，并返回对应的 struct netdev_queue，后续会把这个队列索引传递给驱动，具体实现如下所示：

上述代码的核心就是在复用已有队列和重新选队列之间做决策，优先从 socket 里取之前缓存的发送队列（sk_tx_queue_get），如果这个队列合法且允许继续用，就直接复用，否则（比如第一次发送、队列越界、或允许乱序 ooo_okay）就重新选择，优先尝试用 XPS，选不到再用 flow hash，如果选出的新队列和旧的不一样，还会把结果缓存回 sk，方便后续复用，最终返回 queue_index。

上述代码实现的是 XPS（Transmit Packet Steering）选队列逻辑，如果系统没有开启 XPS，直接返回 ，否则优先走 RXQS 模式（如果开启了 xps_rxqs_needed），尝试根据 sk 的接收队列去匹配一个对应的 TX 队列，实现收发队列对齐(这里分析代码感觉大概率是返回-1)，再退化为 按 CPU 选队列（XPS_CPUS），根据发送的cpu找到该 CPU 绑定的 TX 队列，最终通过 __get_xps_queue_idx 从用户配置的 xps_maps 中选出一个队列。

上述代码的核心逻辑是先确定属于哪一类流量(根据skb的优先级进行映射)+ 当前来源（CPU 或 RX 队列），然后在这类流量允许使用的一组 TX 队列里挑一个。如果这组里只有一个队列就直接用，如果有多个，就用数据包的 hash 算一个，保证同一条流大概率走同一个队列。

skb_tx_hash核心思想是在一组 TX 队列里用 hash 算法选一个具体队列。如果设备启用了 TC（注意这里应该是驱动注册的），会先根据 skb->priority 找到对应的 TC，并把范围限制在这个 TC 对应的一部分队列（起始位置加偏移），如果这个包之前记录过 RX 队列（同源同宿吧，但是大概率应该没有），就直接用 RX 队列号映射到 TX 队列；否则就用 skb_get_hash() 对数据包做 hash来选择队列。

skb_get_hash的作用就是给 skb 计算一个基于流的 hash 值。如果 skb 还没有 hash（这里需要注意，对于tcp来说，在建立连接三次握手的过程种就设置了这个值，因此不会调用__skb_get_hash），就调用 __skb_get_hash()，先用 skb_flow_dissect_flow_keys() 从包里解析出五元组等关键信息（IP、端口、协议等），然后通过 __flow_hash_from_keys() 用 siphash 算法计算一个 hash 值，最后写回到 skb->hash 里。
