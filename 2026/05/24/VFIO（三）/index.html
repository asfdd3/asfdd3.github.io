vfio_iommu_type1（一）

发表于 2026-05-24 分类于 VFIO 阅读次数： Valine：

VFIO 的 Type1 IOMMU 后端，也就是用户态程序通过 VFIO 把设备直通给虚拟机/DPDK 这类程序时，用来管理 IOVA → 物理页 PFN 映射关系的那一层。它不是普通网卡驱动，也不是 vfio-pci 设备驱动，而是 VFIO container 选择 IOMMU 后端后真正负责 DMA 映射、解除映射、pin 用户页、attach group、dirty page tracking 的核心代码。

在分析 vfio_iommu_type1 的具体流程之前，需要先把几个核心数据结构捋清楚。因为这份代码本质上不是在驱动某一个具体 PCI 设备，而是在 VFIO container 这一层维护 IOMMU domain、IOMMU group、用户态 DMA 映射、IOVA 可用范围、pin 住的物理页 之间的关系。

1. struct vfio_iommu

struct vfio_iommu 是这份代码里最核心的结构体，可以把它理解成 一个 VFIO container 对应的 Type1 IOMMU 私有数据。

当用户态对 container 执行 VFIO_SET_IOMMU，选择 Type1 IOMMU 后端时，内核会创建一个 vfio_iommu 对象。后续这个 container 下 attach 进来的 group、用户态 map 进来的 DMA 区间、IOMMU domain、dirty page tracking 等信息，都会挂在这个结构体下面。

第一类是 domain 管理相关字段：

domain_list 里面挂的是 struct vfio_domain。每个 vfio_domain 里面又封装了一个真正的 struct iommu_domain。当 VFIO group attach 到 container 时，就会创建或者复用一个 IOMMU domain，然后挂到这个链表上。

第二类是 DMA 映射管理相关字段：

dma_list 是一棵红黑树，里面挂的是一个个 struct vfio_dma。每个 vfio_dma 表示用户态通过 VFIO_IOMMU_MAP_DMA 注册进来的一段 DMA 映射区间。

所以 dma_list 本质上就是当前 container 已经建立的 DMA 映射表。

第三类是 IOVA 可用范围管理：

iova_list 记录当前 container 允许用户使用的 IOVA 地址范围。这个范围不是用户随便定的，它需要综合 IOMMU domain 的 aperture 以及设备 reserved region。

也就是说，用户态传入的 map->iova 必须落在 iova_list 允许的范围内，否则不能建立 DMA 映射。

第四类是 页大小和 dirty tracking 相关字段：

pgsize_bitmap 表示当前所有 IOMMU domain 共同支持的页大小。因为一个 container 里可能有多个 domain，所以最终能使用的页大小必须是它们的交集。

dirty_page_tracking 表示当前是否开启脏页追踪。这个主要服务于虚拟机热迁移场景，用来记录哪些页被设备 DMA 写脏了。

所以 vfio_iommu 可以总结成一句话：

vfio_iommu 是 Type1 IOMMU 后端的总控结构，它把 container 下的 domain、group、DMA 映射、IOVA 范围、dirty bitmap 等状态统一管理起来。

iova

这是设备 DMA 时使用的地址，也就是设备眼里的地址。

设备并不知道用户态虚拟地址 vaddr，它只知道 IOVA。比如用户态可以告诉 VFIO：

之后设备 DMA 到 0x100000000，IOMMU 就会把它翻译到实际的 host 物理页。

vaddr

这是用户进程的虚拟地址。VFIO 需要根据这个地址找到对应的物理页，然后 pin 住。

所以 vaddr 是给内核查用户页表用的，不是给设备用的。

size

表示这段 DMA 映射的大小。

注意，vfio_dma 在插入红黑树时可能先以 size 为 0 的方式插入，然后在 pin/map 成功后逐步增长。这么做是为了在映射过程中也能保持状态可追踪，出错时可以回滚。

iommu_mapped

表示这段 vfio_dma 是否已经真正映射进 IOMMU 页表。

有些场景下，container 里还没有真实 IOMMU domain，此时 vfio_dma 可能只是先记录下来，还没有真正调用 iommu_map()。

等后面 group attach 进来，有了 domain，再通过 replay 把已有的 DMA 映射同步到新的 domain 中。

task 和 mm

这两个字段用于记录这段 DMA 映射属于哪个进程地址空间。

因为 VFIO pin 用户页时需要做 locked memory 统计，也就是检查用户有没有超过 RLIMIT_MEMLOCK。所以它需要保存对应的 task 和 mm。

pfn_list

这个红黑树用于记录外部 pin 的页。比如 mdev 或 emulated IOMMU 场景下，设备驱动可能通过 VFIO 的 pin_pages 接口单独 pin 某些 IOVA 对应的页，这些页就会记录到 pfn_list 中。

bitmap

这个是 dirty page tracking 用的 bitmap。

当开启脏页追踪后，每个 vfio_dma 会分配一个 bitmap，用来记录这段 DMA 区间内哪些页被认为是 dirty。

所以 vfio_dma 可以总结成一句话：

vfio_dma 表示用户态注册的一段 DMA 区间，它把用户虚拟地址、设备 IOVA、映射大小、访问权限、pin 页信息和 dirty bitmap 绑定在一起。
