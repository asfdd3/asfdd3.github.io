vfio_iommu_type1（三）

发表于 2026-05-28 更新于 2026-05-27 分类于 VFIO 阅读次数： Valine：

在前面的 VFIO 流程中，用户态程序首先会打开 /dev/vfio/vfio 得到一个 container 文件描述符，然后通过 VFIO_SET_IOMMU 给这个 container 绑定一个具体的 IOMMU 后端，例如 VFIO_TYPE1_IOMMU。从这个时刻开始，这个 container 后续收到的 IOMMU 相关 ioctl 请求，就不再只是停留在 VFIO container 层，而是会被转发到 Type1 IOMMU 后端中处理。

真正负责 DMA 映射、查询 IOMMU 能力、等工作的，是后端注册的 vfio_iommu_driver_ops。在 Type1 IOMMU 场景下，这个后端处理函数就是 vfio_iommu_type1_ioctl()。

vfio_dma_do_map() 是 VFIO_IOMMU_MAP_DMA 的核心处理函数。它接收用户态传入的 vaddr、iova 和 size，先检查参数是否溢出、是否页对齐、地址范围是否回绕，然后根据 READ/WRITE 标志设置设备 DMA 的访问权限。接着它会检查这段 IOVA 是否已经存在映射、是否落在允许的 DMA 窗口内（这里注意iova的范围是由iommu的geometry决定的），以及 DMA 映射条目是否还有额度。检查通过后，函数会分配并初始化一个 struct vfio_dma，用它记录这次映射的 iova、vaddr、权限、进程 mm、task 等信息，并把它插入 VFIO 的 DMA 红黑树中。最后，如果 container 已经绑定了 IOMMU domain，就继续调用 vfio_pin_map_dma()，完成用户页 pin 和 IOVA -> 物理页 的 IOMMU 映射。

vfio_pin_map_dma() 是在 vfio_dma_do_map() 创建好 struct vfio_dma 记录之后，继续完成 用户页 pin + IOMMU 映射 的函数。它从 dma->iova 和 dma->vaddr 开始，按照 map_size 一段一段处理用户态内存：每轮先调用 vfio_pin_pages_remote()，从当前 vaddr + dma->size 位置开始，把一批连续用户页 pin 住，并拿到起始物理页帧号 pfn 和连续页数 npage；然后调用 vfio_iommu_map(封装了iommu_map)，把这批物理页映射到设备可见的 IOVA 地址 iova + dma->size 上，也就是建立 IOVA -> PFN 的 IOMMU 页表映射。映射成功后，就减少剩余 size，并增加 dma->size，表示这段 DMA 区域已经成功映射了多少字节。

vfio_pin_pages_remote() 的核是从用户态虚拟地址 vaddr 开始，pin 住一段物理页连续的内存，并返回起始 PFN 和连续页数**。它先通过 vaddr_get_pfns() 获取并固定一批用户页，然后逐页检查 PFN 是否连续，以及页类型是否一致，比如不能把普通用户页和 reserved page 混在同一个连续块里。如果遇到物理页不连续、页类型变化，或者 batch 用完，就停止本轮处理，把已经连续的这一段返回给上层。

同时，它还会对普通用户页做 RLIMIT_MEMLOCK 检查，防止用户 pin 太多内存；如果出错，就把已经 pin 的页释放掉。整体来说，它就是给后面的 vfio_iommu_map() 准备一段 已经 pin 住、PFN 连续、可以直接建立 IOMMU 映射 的物理页区间。
