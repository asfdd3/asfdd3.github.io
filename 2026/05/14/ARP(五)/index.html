neigh_output() 是邻居子系统统一的发包入口，它会根据当前 neighbour 的状态决定 skb 如何发送。如果 neighbour 当前处于 NUD_CONNECTED 状态，并且已经缓存好了二层头部 hh_cache，那么就直接走 neigh_hh_output() 快速发送路径（通常应该走这里），否则就调用 neighbour 的 output 函数。对于 ARP 场景来说，这里的 output 通常会指向 neigh_resolve_output()，它会进一步判断当前是否已经解析出 MAC 地址，如果没有解析完成，就触发 ARP 请求并把 skb 挂入 arp_queue 等待后续发送（前文已经分析过）。

ARP报文的接收

协议栈收到数据包后，若 EtherType 为 ETH_P_ARP (0x6)，内核就会将该 skb 上送给 ARP 模块注册的包处理函数 arp_rcv()。这个函数可以说是 Linux ARP 子系统真正意义上的入口，它负责对 ARP 报文进行合法性校验、过滤异常场景、检查设备状态，并最终调用 arp_process() 对 ARP 请求或响应进行处理。

arp_rcv() 是 Linux 内核 ARP 模块的收包入口，当网卡收到 ETH_P_ARP 类型的数据包后最终会进入这里。函数首先过滤一些不需要处理的报文，比如设备关闭 ARP、发给其他主机的包或回环包，然后通过 skb_share_check() 保证 skb 可写。接着利用 pskb_may_pull() 确保 ARP 头已经在线性区，并检查硬件地址长度和协议地址长度是否合法。校验通过后，清空邻居子系统控制块 NEIGH_CB(skb)，最后通过 NF_HOOK() 先经过 Netfilter 的 NF_ARP_IN 钩子链，如果没有被丢弃，就进入真正的 ARP 处理函数 arp_process()。

arp_process() 是真正处理 ARP 报文的核心函数。它先从 skb 中取出接收设备、ARP 头、源 MAC、源 IP、目标 MAC、目标 IP，然后根据设备类型检查 ARP 报文是否合法，只处理 ARP request 和 ARP reply。接着会过滤掉一些不应该处理的目标地址，比如组播地址、127 回环地址，以及被配置为丢弃的免费 ARP。对于 sip == 0 的特殊 ARP 请求，它通常用于 IPv4 地址冲突检测，如果目标 IP 是本机地址并且允许响应，就直接回复 ARP。普通 ARP 请求会先根据目标 IP 做一次路由查找，如果发现目标 IP 是本机地址，就根据 arp_ignore、arp_filter 等策略决定是否回复，并可能把请求方的 IP/MAC 学习到邻居表；如果目标不是本机，但设备开启了转发并满足代理 ARP 条件，则可能代替别人回复。最后，对于 ARP reply 或其它可学习的 ARP 报文，内核会查找或按配置创建邻居表项，然后调用 neigh_update() 用报文里的源 MAC 更新邻居缓存状态，可能标记为 NUD_REACHABLE 或 NUD_STALE。
