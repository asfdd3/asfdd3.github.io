VFIO框架初始化

发表于 2026-05-21 分类于 VFIO 阅读次数： Valine：

前面我们知道，VFIO 在用户态主要是通过 /dev/vfio/vfio 和 /dev/vfio/<group_id> 这类文件对外提供接口。用户程序通过 open() 拿到 fd，再通过 ioctl() 完成 container、group、device 之间的关联，最后实现设备直通、DMA 映射以及 BAR 空间访问。

vfio_init() 是 VFIO core 模块加载时的初始化入口。这个函数本身不会直接操作某个 PCI 设备，也不会建立 DMA 映射，而是先初始化 VFIO core 运行需要的基础设施。

因此，本文就从 vfio_init() 这个函数开始，沿着它的初始化顺序，分析 VFIO 在模块加载阶段到底创建了哪些对象，以及这些对象如何为后面的设备直通流程做准备。

这段 vfio_init() 是 VFIO core 模块加载时的初始化入口，它的作用不是直接操作某个 PCI 设备，也不是建立 DMA 映射，而是先把 VFIO 框架运行所需的基础设施搭起来。函数一开始通过 ida_init(&vfio.device_ida) 初始化一个 ID 分配器，后续每注册一个 VFIO device 时，都可以从这里分配一个唯一的设备编号，比如用于生成 vfioX 这类设备名。接着调用 vfio_group_init() 初始化 VFIO group 相关的管理逻辑，因为 VFIO 仍然以 IOMMU group 作为重要的隔离和管理边界；然后调用 vfio_virqfd_init() 初始化 irqfd 相关的清理工作队列，用于处理 VFIO 中断事件通知相关的异步清理任务。之后通过 class_create("vfio-dev") 创建一个名为 vfio-dev 的设备类，对应后续 sysfs 中的 /sys/class/vfio-dev/vfioX，也就是说每个 VFIO device 都可以作为一个独立设备挂到这个 class 下面。

vfio_fops_unl_ioctl() 是 /dev/vfio/vfio 这个 container fd 的 ioctl 总入口。用户态对 container_fd 调用 ioctl() 时，都会先进入这个函数。然后根据 cmd 区分不同操作：如果是 VFIO_GET_API_VERSION，就直接返回 VFIO API 版本；如果是 VFIO_CHECK_EXTENSION，就检查当前 container 是否支持某种能力，比如是否支持 VFIO_TYPE1_IOMMU；如果是 VFIO_SET_IOMMU，就调用 vfio_ioctl_set_iommu() 给当前 container 绑定一个具体的 IOMMU 后端。剩下那些 VFIO core 不认识的 ioctl，会走 default 分支，转发给当前 container 已经绑定的 IOMMU driver 处理，比如 DMA map 这类操作。

vfio_ioctl_set_iommu() 的作用就是 给当前 container 绑定一个 IOMMU 后端。它先检查 container 里必须已经有 group，并且还没有设置过 IOMMU；然后遍历 VFIO 已注册的 IOMMU driver，找到支持用户传入类型的后端，比如 VFIO_TYPE1_IOMMU。找到后调用 driver->ops->open(arg) 创建后端私有数据，再通过 __vfio_container_attach_groups() 把 container 里已有的 group attach 到这个 IOMMU 后端。最后把 driver 和 data 保存到 container->iommu_driver、container->iommu_data 中，后续 DMA map/unmap 等 ioctl 就可以转发给这个后端处理。重点就是：选择 IOMMU 后端、open 后端、attach group、保存到 container。

注意：这里遍历的 IOMMU 后端不是临时创建的，而是在对应 IOMMU backend 模块初始化时提前注册到 VFIO core 里的。
