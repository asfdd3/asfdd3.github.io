vfio_iommu_type1（二）

发表于 2026-05-26 更新于 2026-05-25 分类于 VFIO 阅读次数： Valine：

上面文章介绍完关键vfio_iommu_type1关键的数据结构后，本文就从 vfio_iommu_driver_ops_type1 这张回调函数集开始，分析 vfio-iommu-type1 是如何注册、如何在用户执行 VFIO_SET_IOMMU 时被选中、如何 attach group，以及后续如何通过 VFIO_IOMMU_MAP_DMA 建立 DMA 映射关系的。

vfio-iommu-type1 本质上是 VFIO core 的一个 IOMMU 后端实现。它并不是用户态直接打开的设备文件，而是作为一个内核模块，把自己的一组回调函数注册到 VFIO框架中，在VFIO_SET_IOMMU会遍历注册的后端，这时如果用户指定了vfio-iommu-type1就会和对应的container关联起来。

这里最重要的是 vfio_register_iommu_driver()。它会把 vfio_iommu_driver_ops_type1 注册到 VFIO core 维护的 IOMMU 后端驱动链表中，具体代码所示

vfio_register_iommu_driver() 的作用就是把某个 IOMMU 后端驱动的回调表 ops 封装成 vfio_iommu_driver，检查不能重复注册后，挂到 VFIO 全局的 iommu_drivers_list 链表里，后面用户执行 VFIO_SET_IOMMU 时，VFIO core 就会遍历这个链表找到合适的 IOMMU 后端。

对于vfio_iommu_type1来说注册给 VFIO core 的回调函数集如下：

  • 用户态设置 container 的 IOMMU 类型时，会走到 .open；
  • group 加入 container 后，需要绑定到 IOMMU 后端时，会走到 .attach_group；
  • 用户态执行 VFIO_IOMMU_MAP_DMA 或 VFIO_IOMMU_UNMAP_DMA 时，会走到 .ioctl；

vfio_iommu_type1_open() 是 type1 IOMMU 后端的创建入口，它会为当前 VFIO container 分配一个 vfio_iommu 对象，并初始化 domain、IOVA、DMA 映射和设备链表等管理结构，后续 group attach 和 DMA map 都会基于这个对象进行。

vfio_iommu_type1_attach_group

vfio_iommu_type1_attach_group() 负责把 VFIO group 绑定到 type1 后端管理的 iommu_domain 上，是设备正式进入 VFIO DMA 隔离地址空间的关键步骤。

vfio_iommu_type1_attach_group() 的核心作用就是把一个 VFIO group 接入当前 container 的 type1 IOMMU 后端管理中。它先检查这个 group 是否已经 attach 过，然后为它创建 VFIO 自己的 vfio_iommu_group 包装结构；如果是真实 IOMMU group，就再分配一个 vfio_domain，通过 group 里的设备申请 iommu_domain，然后调用 iommu_attach_group() 把这个 group 真正挂到 IOMMU domain 上。接着它会检查这个 domain 支持的 IOVA 地址范围、reserved region、MSI 保留区域、中断隔离是否安全，并尝试复用已有 compatible domain；如果不能复用，就把已有 DMA 映射 replay 到新 domain 里，最后把 group/domain 挂到 iommu->domain_list，更新当前 container 的可用 IOVA 范围。简单说，这个函数就是：为新加入 container 的 group 找一个合适的 IOMMU domain，并保证这个 group 后续 DMA 访问受 type1 IOMMU 地址空间控制。
