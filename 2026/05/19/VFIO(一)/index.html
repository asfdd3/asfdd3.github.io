VFIO 原理

发表于 2026-05-19 更新于 2026-05-18 分类于 VFIO 阅读次数： Valine：

1. VFIO 要解决什么问题

VFIO 可以理解为 Linux 内核提供的一套“安全设备直通框架”。它的目标是把真实硬件设备暴露给用户态程序使用，同时又尽量保证系统内存不会被设备随意访问。

在传统内核驱动模型中，硬件设备通常由内核驱动控制。比如网卡由内核网卡驱动接管，磁盘控制器由块设备驱动接管，GPU 或加速卡也有各自的内核驱动。用户态程序如果想使用这些设备，一般需要通过内核提供的系统调用、字符设备、网络协议栈或者其他内核接口间接访问。

但是在一些高性能场景下，这种方式会带来额外开销。比如虚拟机设备直通、高性能网络转发、用户态驱动、加速卡访问等场景，希望用户态程序能够尽可能直接地操作硬件设备。这样可以减少内核协议栈或虚拟化层带来的中间开销，提高 I/O 性能。

VFIO 的作用就是在“用户态直接访问设备”和“系统安全隔离”之间做一个平衡：

  • 用户态程序可以拿到设备对应的文件描述符；
  • 用户态程序可以通过 ioctl() 查询设备信息；
  • 用户态程序可以通过 mmap() 映射设备 BAR 空间；
  • 用户态程序可以配置 DMA 映射，让设备可以访问指定的用户态内存；
  • 用户态程序可以通过 eventfd 等机制接收设备中断；
  • 内核通过 IOMMU 约束设备 DMA 访问范围，避免设备随意读写系统内存。

一句话说，VFIO 不是单纯地“把设备交给用户态”，而是在 IOMMU 保护下，把设备安全地交给用户态程序或虚拟机使用。

2. VFIO 的典型使用场景

VFIO 最常见的使用场景是虚拟机设备直通，也就是 passthrough。

例如一台物理服务器上有一块 PCIe 网卡，宿主机可以把这块网卡从原来的内核驱动解绑，然后绑定到 vfio-pci 驱动。之后 QEMU/KVM 这类虚拟机进程就可以通过 VFIO 获取这个设备的访问权限，并把它直通给虚拟机。

从虚拟机内部看，这块设备就像插在虚拟机里一样。虚拟机可以加载自己的设备驱动，比如加载网卡驱动，并直接操作这个 PCI 设备。这样虚拟机的数据收发可以绕过传统的虚拟网卡模拟路径，获得更低延迟和更高带宽。

除了虚拟机直通，VFIO 也可以用于用户态驱动。比如某些高性能计算程序、用户态网络程序、加速卡控制程序，希望在用户态直接操作设备寄存器和 DMA 队列，也可以使用 VFIO。相比早期的 UIO 方式，VFIO 最大的优势是引入了 IOMMU 隔离和更完整的中断、DMA、设备 region 管理能力。

3. 为什么 VFIO 必须依赖 IOMMU

设备直接访问内存最危险的地方在于 DMA。

CPU 访问内存时会经过 MMU，进程只能访问自己的虚拟地址空间。但是硬件设备做 DMA 时，如果没有额外限制，设备可能直接访问物理内存。一旦用户态程序可以控制设备 DMA 地址，而系统又没有隔离机制，那么这个设备理论上就可能读写任意物理内存，这会破坏整个系统安全。

IOMMU 的作用类似于设备侧的 MMU。它负责把设备看到的 DMA 地址转换成真实物理地址，并且可以限制设备只能访问被允许的内存范围。

在 VFIO 中，用户态程序并不是随便给设备一个物理地址，而是先通过 VFIO 接口建立 DMA 映射：

这里的含义是：用户态申请了一段虚拟内存，然后告诉 VFIO，把这段用户态内存映射到设备视角下的 IOVA 地址。之后设备进行 DMA 时，访问的是 IOVA，而 IOMMU 会把这个 IOVA 翻译到对应的真实物理页。

所以 VFIO 的 DMA 安全性主要来自两点：

  1. 设备不能直接访问任意物理地址；
  2. 设备只能访问用户态显式映射给它的内存范围。

这也是 VFIO 比 UIO 更适合设备直通和用户态驱动的重要原因。

4. VFIO 的三个核心对象：Container、Group、Device

VFIO 里最重要的三个概念是：

它们分别表示不同层次的管理对象。

4.1 Device：真正要操作的硬件设备

Device 就是用户态最终要访问的硬件设备，比如一个 PCI 网卡、GPU、NVMe 控制器或者其他 PCIe 设备。

设备本身会提供几类资源：

  • 配置空间；
  • BAR 空间；
  • 中断；
  • DMA 能力；
  • 设备复位能力；
  • 其他设备私有操作。

通过 VFIO，用户态可以拿到一个 device fd。拿到这个 fd 后，用户态就可以对设备执行 VFIO_DEVICE_GET_INFO、VFIO_DEVICE_GET_REGION_INFO、VFIO_DEVICE_GET_IRQ_INFO、VFIO_DEVICE_RESET 等操作。

也就是说，真正操作硬件设备时，最终落到的是 device fd 上。

4.2 Group：VFIO 的安全隔离单位

VFIO 不是以单个设备作为最小安全单位，而是以 IOMMU Group 作为最小安全单位。

这是因为硬件隔离粒度不一定能做到单设备级别。比如：

  • 一个多功能 PCI 设备的多个 function 之间可能存在内部通道；
  • 某些 PCIe bridge 不支持 ACS，导致桥后设备之间的访问不一定经过 IOMMU；
  • PCIe-to-PCI bridge 后面的多个设备在 IOMMU 看来可能无法精确区分；
  • 某些平台的 IOMMU 拓扑本身只能以更粗粒度隔离设备。

因此，内核会把“不能彼此安全隔离”的设备放到同一个 IOMMU Group 里。

VFIO 要求：如果用户态想使用某个 Group，那么这个 Group 里的所有设备都必须处于安全可控状态。通常要求这些设备都绑定到 VFIO 驱动，或者至少没有被宿主机普通驱动占用。否则 VFIO 会认为这个 Group 不可用。

这就是为什么有时你只想直通一个 PCI 设备，却发现同一个 IOMMU Group 里还有其他设备也必须处理。因为 VFIO 的安全边界不是你想直通的那个单独设备，而是整个 IOMMU Group。

Container 是 VFIO 中用来承载 IOMMU 上下文的对象。用户态通过打开 /dev/vfio/vfio 创建一个 container：

7. Device fd 上可以做什么

拿到 device fd 后，用户态程序可以做几类事情。

7.1 查询设备基本信息

这个接口会返回设备支持的 region 数量、中断数量、设备能力标志等信息。

7.2 查询设备 Region

Region 可以理解为设备暴露给用户态的可访问区域。对于 PCI 设备来说，常见 region 包括：

  • PCI 配置空间；
  • BAR0；
  • BAR1；
  • 其他 BAR；
  • 设备特定区域。

用户态可以根据 region 信息决定如何访问这些区域。某些 region 可以通过 read/write 访问，某些 region 可以通过 mmap() 映射到用户态地址空间。

8. VFIO 和虚拟机设备直通的关系

当你在宿主机上创建虚拟机，并把某个 PCI 设备 passthrough 给虚拟机时，底层通常就是 VFIO 在工作。

以 QEMU/KVM 为例，用户配置设备直通后，QEMU 进程会通过 VFIO：

  1. 打开 /dev/vfio/vfio 创建 container；
  2. 打开 /dev/vfio/$GROUP 获取 group fd；
  3. 检查 Group 是否 viable；
  4. 把 Group 加入 Container；
  5. 设置 IOMMU；
  6. 获取 device fd；
  7. mmap 设备 BAR；
  8. 配置中断；
  9. 把设备资源呈现给虚拟机；
  10. 虚拟机里的驱动像操作真实硬件一样操作这个设备。

所以，从宿主机视角看，QEMU 进程就是一个用户态设备驱动。从虚拟机视角看，设备就像真的插在虚拟机里。

这也是 VFIO 文档中说的关键点：设备直通会把虚拟机进程变成一个用户态驱动。它的收益是更低延迟、更高带宽，以及虚拟机内部可以直接使用裸机设备驱动。

9. VFIO 和 UIO 的区别

UIO 也可以把设备部分能力暴露给用户态，但 UIO 的安全和功能都比较有限。

VFIO 相比 UIO 的主要优势是：

对比点 UIO VFIO
DMA 安全隔离 较弱，缺少完整 IOMMU 管理 依赖 IOMMU，安全性更强
PCI 配置空间访问 能力有限 支持更完整的设备访问模型
中断支持 相对简单 支持更完整的中断描述和 eventfd 通知
设备直通 不适合复杂虚拟化场景 是 KVM/QEMU 设备直通的核心机制
安全边界 主要依赖权限控制 以 IOMMU Group 作为隔离单位
用户态驱动能力 基础 更完整、更规范

所以，如果只是简单实验设备寄存器访问，UIO 可能够用；但如果涉及 DMA、虚拟机直通、高性能用户态驱动，VFIO 更合适。

10. 从 fd 的角度理解 VFIO

VFIO 很适合从文件描述符的角度理解。

10.1 Container fd

这个 fd 代表一个 VFIO container，也就是一套 DMA/IOMMU 管理上下文。

它主要负责：

  • 查询 VFIO API 版本；
  • 检查 IOMMU 能力；
  • 启用 IOMMU；
  • 建立和解除 DMA 映射。

10.2 Group fd

这个 fd 代表一个 VFIO Group。

它主要负责：

  • 查询 Group 是否 viable；
  • 把 Group 加入 Container；
  • 从 Group 中获取具体 Device fd。

10.3 Device fd

这个 fd 代表一个具体设备。

它主要负责：

  • 查询设备信息；
  • 查询 region；
  • 访问设备 region；
  • 配置中断；
  • 复位设备；
  • 执行设备私有 ioctl。

11.1 init / release

init 和 release 分别在 VFIO device 初始化和释放时调用。驱动可以在这里初始化自己的私有状态，或者释放设备相关资源。

11.2 open_device / close_device

open_device 在用户态第一次获取 device fd 时调用。比如用户态通过 VFIO_GROUP_GET_DEVICE_FD 获取设备 fd 时，如果这是该设备当前会话中的第一次打开，VFIO core 会调用设备驱动的 open_device。

close_device 则在用户态关闭设备 fd 后执行，用于释放会话相关资源。

11.3 read / write / mmap

这些回调用于实现设备 region 的访问。

如果用户态通过 read() 或 write() 访问某个设备 region，最终可能会进入驱动的 read/write 回调。

如果用户态通过 mmap() 映射设备 region，比如映射 PCI BAR 空间，则会进入 mmap 回调。

11.4 ioctl

ioctl 用于处理设备相关控制命令。VFIO core 会处理通用命令，而某些设备相关命令会透传给具体设备驱动。

11.5 request

request 通常用于设备即将被 unregister、解绑或需要用户态释放设备时通知用户态。比如宿主机希望回收设备，VFIO 可以通过 request 机制通知当前使用设备的用户态进程。

11.6 dma_unmap

当某段 IOVA 被解除映射时，如果驱动使用了 VFIO 页固定相关接口，就需要通过 dma_unmap 回调释放或同步对应资源。

这个回调需要能处理比较早期的调用场景，也就是即使设备还没有被用户态打开，也要能安全处理。

13. 总结

VFIO 的核心思想是：把设备直接访问能力交给用户态，但不放弃内核对安全边界的控制。

它通过 IOMMU Group 保证设备隔离边界，通过 Container 管理 DMA 地址空间，通过 Device fd 暴露具体设备能力。用户态程序拿到设备 fd 后，可以访问设备 region、配置中断、建立 DMA 映射，从而实现高性能设备直通或用户态驱动。

对于虚拟机 passthrough 来说，VFIO 是宿主机把真实 PCI 设备安全交给 QEMU/KVM 的关键机制。对于用户态驱动来说，VFIO 提供了比 UIO 更完整、更安全的设备访问模型。
